# Oracle harness

The "oracle" is the original Windows Speechify 3.0.5 engine, accessed via
[../../bin/spfy_dumpwav.exe](../../../bin/spfy_dumpwav.exe). All
acceptance tests for the CORE track (M0b through M5) compare native
output against WAVs generated by this oracle on a fixed corpus.

## Files

- [corpus.jsonl](corpus.jsonl) — input phrases, one JSON object per line.
  Each entry has `id`, `mode` (`text` or `spr`), `text`, `tags`.
- [run_corpus.py](run_corpus.py) — drives the oracle exe over every
  corpus entry, writes `wavs/<id>.wav`. **Run once on Windows**, commit
  the WAVs.
- `wavs/` — committed oracle WAVs (or stored via git-lfs if too large).

## Generating oracle WAVs

```sh
# 1. Start the Speechify server (oracle exe is a client; needs the server)
#    e.g. via BalRegisterVoice.bat or however the user starts Speechify.exe.
# 2. From the project root:
"C:\Program Files\Python312\python.exe" spfy/test/oracle/run_corpus.py
```

The script uses `bin/spfy_dumpwav.exe` by default. Override with
`--dumpwav PATH`. Filter to a subset via `--filter REGEX` on the entry id.

## Comparing native output against the oracle

Once the native build produces WAVs into `build/x87/native_wavs/`, diff
against the oracle:

```sh
python spfy/test/diff/wav_diff.py build/x87/native_wavs/ \
                                  spfy/test/oracle/wavs/ --strict
# acceptance bar (M5): 200/200 byte-exact match. 0 divergences.
```

## Corpus stratification

The plan targets ~200 phrases in four buckets:
- 50 single-phoneme sanity (`.0a`, `.1k`, ...)
- 50 minimal-pair voiced/unvoiced joins (WSOLA stress test)
- 50 random natural phrases
- 50 edge cases (long pauses, all-vowel sequences, all-consonant clusters)

The committed seed corpus is a smaller stratified set; expand it
opportunistically as divergences are found during M0b–M4.
